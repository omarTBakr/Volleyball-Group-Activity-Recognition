\begin{abstract}
   This report documents the data pipeline, loader architecture, and first experimental results for the Volleyball Group Activity Recognition project. The dataset, introduced in the CVPR 2016 paper by Mostafa S. Ibrahim et al., contains 55 volleyball videos with two annotation levels: 8 scene-level group activities and 9 player-level individual actions. We describe the two-stage parsing pipeline that unifies three raw annotation sources into a single fast-loading pickle cache, and the generic PyTorch data loader that supports all eight baseline models (B1--B8). We present two completed baselines: \emph{Baseline 1} (B1), a two-stage fine-tuned ResNet-50 single-frame classifier; and \emph{Baseline 3} (B3), a person-then-group architecture that first trains a ResNet-50 on player crops for the 9 person-action labels, then freezes that backbone and learns an MLP on max+mean-concat-pooled per-player features to predict the 8 group activities. Both baselines are reported with training configuration and test-set evaluation.
\end{abstract}

\section{Introduction}

    \taustart{V}olleyball Group Activity Recognition is a hierarchical classification problem where the goal is to classify the group activity of a volleyball scene (one of 8 classes) while also recognising individual player actions (one of 9 classes).
    The dataset was introduced in the CVPR 2016 paper \href{https://www.cs.sfu.ca/~mori/research/papers/ibrahim-cvpr16.pdf}{``A Hierarchical Deep Temporal Model for Group Activity Recognition''} by Mostafa S. Ibrahim et al.
    You can download the dataset from \href{https://github.com/mostafa-saad/deep-activity-rec}{the official repository} 
    or from \href{https://www.kaggle.com/datasets/ahmedmohamed365/volleyball}{Kaggle}.

\section{Raw Data Sources}

The raw dataset provides three separate annotation files per clip:

\begin{enumerate}
    \item \texttt{action\_detections.txt} — Tab-separated file with per-frame action detections. Each line contains the frame name, number of entries, and for each entry: bounding box $(x, y, w, h)$, confidence score, and action label.
    
    \item \texttt{person\_detections.txt} — Same format as action detections but for generic person detection.
    
    \item \texttt{clip\_id.txt} (tracking) — Space-separated file with tracked players across frames. Each line: track ID, bounding box $(x_1, y_1, x_2, y_2)$, frame number, three flag attributes, and action label.
    
    \item \texttt{annotations.txt} (per video) — Contains the group activity label and per-person bounding boxes for each clip's middle frame.
\end{enumerate}

\section{Two-Stage Parsing Pipeline}

We unify the raw annotations into a single data structure through a two-stage pipeline implemented in \texttt{src/json\_parser.py}.

\subsection{Stage 1: Player-Level Parsing}

\texttt{create\_master\_json()} iterates over all 55 videos and their clips, parsing the three annotation files into a unified JSON structure keyed by \texttt{video\_id/clip\_id}. Each clip entry contains:
\begin{itemize}
    \item \texttt{actions}: per-frame action detections $\{$frame $\rightarrow$ list of $\{$box, score, label$\}\}$
    \item \texttt{persons}: per-frame person detections (same format)
    \item \texttt{tracking}: per-frame tracked players $\{$frame $\rightarrow$ list of $\{$id, box, flags, action$\}\}$
\end{itemize}

\subsection{Stage 2: Scene-Level Enrichment}

\texttt{enrich\_with\_scene\_labels()} reads each video's \texttt{annotations.txt} to extract the group-activity label and merges it into the master JSON under a new \texttt{scene\_class} key per clip.

\subsection{Pickle Caching}

The enriched master JSON ($\sim$1.6\,GB) is dumped to a pickle file ($\sim$247\,MB) via \texttt{pickle\_dump.py} for fast loading. The dump function is a \textbf{singleton}: it skips re-dumping if the pickle file already exists.

\section{Data Loader Architecture}

The \texttt{VolleyballDataset} class in \texttt{src/data/data\_loader.py} is a generic PyTorch \texttt{Dataset} that loads from the pickle cache and supports all baseline models through constructor flags.

\subsection{Constructor Parameters}

\begin{itemize}
    \item \texttt{mode}: Dataset split --- \texttt{"train"}, \texttt{"validation"}, or \texttt{"test"}
    \item \texttt{n\_frames}: Number of frames per clip (1 or 9). Must be a positive odd number.
    \item \texttt{full\_image}: Return full-resolution frames (for B1, B4)
    \item \texttt{crop}: Return per-person crops from bounding boxes (for B3, B5--B8)
    \item \texttt{transform}: A torchvision transform pipeline
\end{itemize}

\subsection{Collate Function}

A custom \texttt{collate\_fn} handles the variable number of players per clip by padding the player dimension to the batch maximum and returning a boolean mask tensor indicating valid player positions.

\subsection{Person Boxes}

For crops, the loader uses \textbf{tracking data} (which provides consistent player IDs across frames) as the primary source for bounding boxes, with a fallback to action detections. This is critical for temporal baselines (B5--B8) that need to track the same player across the 9-frame window.

The project implements a progressive series of baseline models, each adding complexity:

\begin{itemize}
    \item \textbf{B1}: Fine-tune ResNet50 on the middle frame as an 8-class image classifier.
    \item \textbf{B3}: Fine-tune ResNet50 on cropped persons (9 action classes); freeze the backbone, max+mean concat-pool all per-player features, train MLP over 8 group classes.
    \item \textbf{B4}: Extract frame-level representations using B1's backbone over 9 frames, train LSTM on the temporal sequence.
    \item \textbf{B5}: LSTM on per-player temporal crops, max-pool player representations, classify with NN.
    \item \textbf{B6}: Pool person features per frame, then LSTM on the image-level sequence.
    \item \textbf{B7}: Full hierarchical model --- LSTM$_1$ per player, max-pool per frame, LSTM$_2$ on the frame sequence.
    \item \textbf{B8}: Same as B7, but pools team 1 (6 players) and team 2 (6 players) separately, then concatenates.
\end{itemize}


\section{Baseline 1: Single-Frame Classifier}

\subsection{Architecture}

Baseline 1 fine-tunes a pretrained ResNet-50 on the middle frame of each clip to directly predict the 8-class group activity label. No temporal or player-level modelling is used; the final fully-connected layer is replaced with a linear head of size 8.

\subsection{Training Strategy}

Training follows a two-stage schedule to stabilise optimisation:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Linear probe} --- backbone frozen, head trained for 5 epochs at $\text{lr} = 10^{-3}$.
    \item \textbf{Full fine-tune} --- entire network unfrozen, up to 50 epochs with differential learning rates (backbone $10^{-4}$, head $3 \times 10^{-4}$) and cosine annealing.
\end{enumerate}

\section{Baseline 3: Person-Then-Group Crop Classifier}

\subsection{Architecture}

Baseline 3 decomposes group activity recognition into two stages, each with its own objective. \textbf{Stage A} treats each player crop as an independent training sample and fine-tunes a ResNet-50 end-to-end to predict its 9-class person action (\texttt{blocking}, \texttt{digging}, \dots, \texttt{waiting}). \textbf{Stage B} loads the trained backbone, replaces its final fully-connected layer with \texttt{Identity} (so it emits a 2048-dim feature vector per crop), freezes all backbone parameters, then for each clip pushes its per-player crops through the backbone to produce a $(P, 2048)$ feature matrix, aggregates across the player dimension by concatenating max-pool and mean-pool to obtain a $(2 \times 2048)$-wide vector, and feeds that to a small MLP head that classifies the 8 group activities.

The dual aggregation is motivated by the side-conditioned nature of the group classes (\texttt{l-spike} vs \texttt{r\_spike}, etc.): max-pool answers \emph{``is any player exhibiting feature $k$ strongly?''} while mean-pool answers \emph{``what is the typical team level of feature $k$?''} The concatenated vector lets the head exploit both signals without committing to one aggregation. Padded player slots are excluded from both pools via the per-clip mask returned by the crop-mode collate function (padded crops are driven to $-\infty$ for max-pool and to $0$ for mean-pool, then mean is divided by the per-clip count of valid players).

\subsection{Training Strategy}

Both stages use SGD with momentum 0.9 and Nesterov, plus inverse-frequency CrossEntropy weights ($w_k = N / (K \cdot n_k)$) computed from the train split's clip metadata.

\begin{enumerate}[leftmargin=*]
    \item \textbf{Stage A (person-action pretraining).} The full ResNet-50 trains on per-crop 9-class CE. Without class weighting the loss collapses on \texttt{standing} (the majority class, $\approx$70\% of all crops) and the backbone never learns to discriminate the minority actions that carry signal for Stage B. With inverse-frequency weights the gradient pull on the rare actions (\texttt{spiking}, \texttt{setting}, \texttt{falling}) is restored to be roughly equal to that on \texttt{standing}.
    \item \textbf{Stage B (group-activity head).} The backbone is loaded from Stage A's best checkpoint, its \texttt{fc} replaced with \texttt{Identity}, and frozen (BatchNorm running stats are held in \texttt{eval} mode via a \texttt{train()} override on the wrapper module so they do not drift). Only the MLP head trains. Class weighting is applied again because the group split is skewed: \texttt{l/r\_winpoint} are $\approx$2.5$\times$ rarer than the spike/pass/set classes.
\end{enumerate}

On multi-GPU hosts (Kaggle's dual-T4 in particular) both stages wrap the model in \texttt{nn.DataParallel} when more than one CUDA device is visible. Checkpoint state-dicts are saved from the unwrapped inner module so they round-trip cleanly into either a wrapped or unwrapped model.

Macro F1 improved from $0.178$ on the legacy max-only-pool run (no class weighting, see commit history) to $0.227$ with the current code, a $\approx 28\%$ relative gain. The improvement comes primarily from recovered recall on the rare \texttt{l/r\_winpoint} classes once their gradient contribution was up-weighted in Stage B's CrossEntropy and the head was given the additional mean-pooled signal alongside the max-pooled one. The evaluation script in \texttt{utils/evaluate.py} auto-detects the saved pool mode from the checkpoint's classifier-input dimension, so the legacy and current checkpoints can both be scored without code changes.

\section{Project Structure}

The codebase is organized as follows:

\begin{itemize}
    \item \texttt{configs/} --- Centralised configuration: paths, splits, label mappings, and per-baseline Hydra YAML configs.
    \item \texttt{src/json\_parser.py} --- Two-stage parsing pipeline.
    \item \texttt{src/pickle\_dump.py} --- Singleton pickle dump/load utility.
    \item \texttt{src/data/kaggle\_data\_loader.py} --- Generic PyTorch Dataset and collate function.
    \item \texttt{src/data/data\_summary.py} --- Dataset statistics and class distributions.
    \item \texttt{models/baseline1.py} --- B1: two-stage fine-tuned ResNet-50 (\textit{complete}).
    \item \texttt{models/baseline3.py} --- B3: person-then-group crop classifier (\textit{complete}).
    \item \texttt{utils/utility.py} --- Training/evaluation loop helpers.
    \item \texttt{utils/evaluate.py} --- Post-training evaluation: loads checkpoint, runs inference, generates all metric plots.
    \item \texttt{utils/plotting.py} --- Confusion matrix, classification report, PR curves, mAP bar chart.
    \item \texttt{utils/load\_model\_config.py} --- Hydra config builders for transforms and scheduler.
\end{itemize}
