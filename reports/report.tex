\begin{abstract}    
   This report documents the data pipeline and loader architecture for the Volleyball Group Activity Recognition project. The dataset, introduced in the CVPR 2016 paper by Mostafa S. Ibrahim et al., contains 55 volleyball videos with two annotation levels: 8 scene-level group activities and 9 player-level individual actions. We describe the two-stage parsing pipeline that unifies three raw annotation sources into a single fast-loading pickle cache, and the generic PyTorch data loader that supports all eight baseline models (B1--B8) through configurable constructor flags.
\end{abstract}

\section{Introduction}

    \taustart{V}olleyball Group Activity Recognition is a hierarchical classification problem where the goal is to classify the group activity of a volleyball scene (one of 8 classes) while also recognising individual player actions (one of 9 classes).
    The dataset was introduced in the CVPR 2016 paper \href{https://www.cs.sfu.ca/~mori/research/papers/ibrahim-cvpr16.pdf}{``A Hierarchical Deep Temporal Model for Group Activity Recognition''} by Mostafa S. Ibrahim et al.
    You can download the dataset from \href{https://github.com/mostafa-saad/deep-activity-rec}{the official repository} 
    or from \href{https://www.kaggle.com/datasets/ahmedmohamed365/volleyball}{Kaggle}.

\section{Raw Data Sources}

The raw dataset provides three separate annotation files per clip:

\begin{enumerate}
    \item \texttt{action\_detections.txt} — Tab-separated file with per-frame action detections. Each line contains the frame name, number of entries, and for each entry: bounding box $(x, y, w, h)$, confidence score, and action label.
    
    \item \texttt{person\_detections.txt} — Same format as action detections but for generic person detection.
    
    \item \texttt{clip\_id.txt} (tracking) — Space-separated file with tracked players across frames. Each line: track ID, bounding box $(x_1, y_1, x_2, y_2)$, frame number, three flag attributes, and action label.
    
    \item \texttt{annotations.txt} (per video) — Contains the group activity label and per-person bounding boxes for each clip's middle frame.
\end{enumerate}

\section{Two-Stage Parsing Pipeline}

We unify the raw annotations into a single data structure through a two-stage pipeline implemented in \texttt{src/json\_parser.py}.

\subsection{Stage 1: Player-Level Parsing}

\texttt{create\_master\_json()} iterates over all 55 videos and their clips, parsing the three annotation files into a unified JSON structure keyed by \texttt{video\_id/clip\_id}. Each clip entry contains:
\begin{itemize}
    \item \texttt{actions}: per-frame action detections $\{$frame $\rightarrow$ list of $\{$box, score, label$\}\}$
    \item \texttt{persons}: per-frame person detections (same format)
    \item \texttt{tracking}: per-frame tracked players $\{$frame $\rightarrow$ list of $\{$id, box, flags, action$\}\}$
\end{itemize}

\subsection{Stage 2: Scene-Level Enrichment}

\texttt{enrich\_with\_scene\_labels()} reads each video's \texttt{annotations.txt} to extract the group-activity label and merges it into the master JSON under a new \texttt{scene\_class} key per clip.

\subsection{Pickle Caching}

The enriched master JSON ($\sim$1.6\,GB) is dumped to a pickle file ($\sim$247\,MB) via \texttt{pickle\_dump.py} for fast loading. The dump function is a \textbf{singleton}: it skips re-dumping if the pickle file already exists.

\section{Data Loader Architecture}

The \texttt{VolleyballDataset} class in \texttt{src/data/data\_loader.py} is a generic PyTorch \texttt{Dataset} that loads from the pickle cache and supports all baseline models through constructor flags.

\subsection{Constructor Parameters}

\begin{itemize}
    \item \texttt{mode}: Dataset split --- \texttt{"train"}, \texttt{"validation"}, or \texttt{"test"}
    \item \texttt{n\_frames}: Number of frames per clip (1 or 9). Must be a positive odd number.
    \item \texttt{full\_image}: Return full-resolution frames (for B1, B4)
    \item \texttt{crop}: Return per-person crops from bounding boxes (for B3, B5--B8)
    \item \texttt{transform}: A torchvision transform pipeline
\end{itemize}

\subsection{Return Shapes}

\begin{table}[h]
    \centering
    \caption{Data loader output shapes by configuration}
    \label{tab:loader-shapes}
    \begin{tabular}{llp{4cm}}
        \toprule
        \textbf{Config} & \textbf{Baseline} & \textbf{Output} \\
        \midrule
        full, $T{=}1$  & B1 & (image, group\_label) \\
        full, $T{=}9$  & B4 & (images $[T,C,H,W]$, group\_label) \\
        crop, $T{=}1$  & B3 & (crops $[P,C,H,W]$, person\_labels $[P]$, group\_label) \\

    \end{tabular}
    \tabletext{$T$: number of frames, $P$: number of players (up to 12), $C$: channels, $H/W$: spatial dimensions.}
\end{table}

\subsection{Collate Function}

A custom \texttt{collate\_fn} handles the variable number of players per clip by padding the player dimension to the batch maximum and returning a boolean mask tensor indicating valid player positions.

\subsection{Person Boxes}

For crops, the loader uses \textbf{tracking data} (which provides consistent player IDs across frames) as the primary source for bounding boxes, with a fallback to action detections. This is critical for temporal baselines (B5--B8) that need to track the same player across the 9-frame window.

\section{Baseline Models Overview}

The project implements a progressive series of baseline models, each adding complexity:

\begin{table}[h]
    \centering
    \caption{Baseline model characteristics}
    \label{tab:baselines}
    \begin{tabular}{lccc}
        \toprule
        \textbf{Model} & \textbf{Temporal} & \textbf{Player} & \textbf{Scene} \\
        \midrule
        B1 & --- & --- & Image clf. \\
        B3 & --- & Crop clf. & Max-pool \\
        B4 & LSTM & --- & LSTM \\
        B5 & LSTM/player & Max-pool & NN \\
        B6 & LSTM/image & Max-pool & LSTM \\
        B7 & LSTM$_1$ + LSTM$_2$ & Max-pool & LSTM$_2$ \\
        B8 & LSTM$_1$ + LSTM$_2$ & Team-split & LSTM$_2$ \\
        \bottomrule
    \end{tabular}
\end{table}

\begin{itemize}
    \item \textbf{B1}: Fine-tune ResNet50 on the middle frame as an 8-class image classifier.
    \item \textbf{B3}: Fine-tune ResNet50 on cropped persons (9 action classes), extract features, max-pool all players, train NN over 8 group classes.
    \item \textbf{B4}: Extract frame-level representations using B1's backbone over 9 frames, train LSTM on the temporal sequence.
    \item \textbf{B5}: LSTM on per-player temporal crops, max-pool player representations, classify with NN.
    \item \textbf{B6}: Pool person features per frame, then LSTM on the image-level sequence.
    \item \textbf{B7}: Full hierarchical model --- LSTM$_1$ per player, max-pool per frame, LSTM$_2$ on the frame sequence.
    \item \textbf{B8}: Same as B7, but pools team 1 (6 players) and team 2 (6 players) separately, then concatenates.
\end{itemize}


\section{Project Structure}

The codebase is organized as follows:

\begin{itemize}
    \item \texttt{configs/} --- Centralised configuration: paths, splits, label mappings, and per-baseline Hydra YAML configs.
    \item \texttt{src/json\_parser.py} --- Two-stage parsing pipeline.
    \item \texttt{src/pickle\_dump.py} --- Singleton pickle dump/load utility.
    \item \texttt{src/data/data\_loader.py} --- Generic PyTorch Dataset and collate function.
    \item \texttt{src/data/data\_summary.py} --- Dataset statistics and class distributions.
    \item \texttt{models/} --- Baseline model implementations (B1, B3, etc.).
    \item \texttt{utils/utility.py} --- Shared training/evaluation helpers.
\end{itemize}
